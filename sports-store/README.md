# Sports Store — Microservices Architecture

Event-driven microservices for a sports e-commerce platform using RabbitMQ, Nodemailer, Winston, and Docker.

---

## Services

| Service | Port | Role |
|---|---|---|
| `order-service` | 3001 | Handles all order CRUD, publishes order events |
| `product-service` | 3002 | Admin product/stock management, publishes inventory events |
| `inventory-service` | — | Consumes events, maintains stock ledger |
| `notification-service` | — | Sends transactional emails via Nodemailer |
| `logger-service` | — | Persists all events to MongoDB via Winston |
| `scheduler-service` | 3005 | Generates and emails reports on admin-set intervals |
| `rabbitmq` | 5672 / 15672 | Message broker (management UI at :15672) |
| `mongo` | 27017 | Shared MongoDB instance (one DB per service) |

---

## Quick Start

### 1. Clone and configure environment

```bash
cp .env .env.local
# Edit .env with your real SMTP credentials and admin email
```

### 2. Build and run all services

```bash
docker compose up --build
```

### 3. Verify RabbitMQ

Open http://localhost:15672 in your browser.
Login: `guest` / `guest`

You should see the `sports_store` topic exchange and all consumer queues after the first event is published.

## Environment Variables

Copy `.env` at the root and fill in:

| Variable | Description |
|---|---|
| `SMTP_HOST` | SMTP server (e.g. `smtp.gmail.com`) |
| `SMTP_PORT` | Usually `587` |
| `SMTP_USER` | Your email address |
| `SMTP_PASS` | App password (not your account password) |
| `ADMIN_EMAIL` | Where reports and low-stock alerts are sent |
| `DEFAULT_REPORT_INTERVAL` | `daily`, `weekly`, or `monthly` |

> For Gmail, generate an App Password at https://myaccount.google.com/apppasswords

## RabbitMQ Event Map

All events flow through one **topic exchange** named `sports_store`.

| Routing Key | Published By | Consumed By |
|---|---|---|
| `order.created` | order-service | inventory, notification, logger |
| `order.status.updated` | order-service / inventory-service | notification, logger |
| `order.cancelled` | order-service | inventory, notification, logger |
| `inventory.product.added` | product-service | inventory, logger |
| `inventory.stock.updated` | product-service | inventory, logger |
| `inventory.stock.low` | product-service / inventory-service | notification, logger |

## API Reference

### Order Service (port 3001)

| Method | Endpoint | Description |
|---|---|---|
| `POST` | `/api/orders` | Place a new order |
| `GET` | `/api/orders` | Get all orders (admin) |
| `GET` | `/api/orders/user/:userId` | Get user's orders |
| `PUT` | `/api/orders/:orderId/status` | Update status (admin) |
| `PUT` | `/api/orders/:orderId/delivery` | Update delivery info |
| `PUT` | `/api/orders/:orderId/cancel` | Cancel an order |

**POST /api/orders — body:**
```json
{
  "userId": "64abc...",
  "userEmail": "customer@example.com",
  "items": [
    { "productId": "64xyz...", "name": "Football", "quantity": 2, "price": 29.99 }
  ],
  "deliveryInfo": { "fullName": "John Doe", "address": "123 Main St", "city": "Dhaka", "phone": "01700000000" },
  "paymentMethod": "card",
  "totalAmount": 59.98
}
```

### Product Service (port 3002)

| Method | Endpoint | Description |
|---|---|---|
| `GET` | `/api/products` | List all products |
| `GET` | `/api/products/:productId` | Get single product |
| `POST` | `/api/products` | Add product (admin) |
| `PUT` | `/api/products/:productId` | Update product (admin) |
| `PATCH` | `/api/products/:productId/stock` | Update stock only (admin) |

**Headers for admin routes:** `x-admin-id: <adminUserId>`

**PATCH /api/products/:id/stock — body:**
```json
{ "stock": 50 }
```

### Scheduler Service (port 3005)

| Method | Endpoint | Description |
|---|---|---|
| `PUT` | `/api/scheduler/interval` | Change report interval |
| `POST` | `/api/scheduler/run-now` | Trigger report immediately |
| `GET` | `/health` | Health check |

**PUT /api/scheduler/interval — body:**
```json
{ "interval": "weekly" }
```

## Database Layout

Each service owns its own MongoDB database — they never query each other's DB directly. All cross-service data flows through RabbitMQ events.

| Database | Owned By | Key Collections |
|---|---|---|
| `sports-store-orders` | order-service | `orders` |
| `sports-store-products` | product-service | `products` |
| `sports-store-inventory` | inventory-service | `stockledgers` |
| `sports-store-logs` | logger-service + scheduler-service | `eventlogs`, `reports`, `reportsettings` |

> `scheduler-service` shares the `sports-store-logs` database so it can query `eventlogs` for report generation without any inter-service HTTP calls.

---

## Report System

Reports are generated by `scheduler-service` using `node-cron`.

- **Default interval:** monthly (configurable via API or `DEFAULT_REPORT_INTERVAL` env var)
- **Run time:** 07:00 UTC (`daily` = every day, `weekly` = every Monday, `monthly` = 1st of month)
- **Delivery:** HTML email sent to `ADMIN_EMAIL`
- **Storage:** each report saved to `reports` collection in `sports-store-logs`

Each report includes: total orders, revenue, cancellations, new products, stock updates, low-stock alerts, and top 5 products by units sold.

## Development Tips

**Watch logs for a specific service:**
```bash
docker compose logs -f order-service
```

**Restart a single service after code change:**
```bash
docker compose up --build order-service
```

**Trigger a test report without waiting for the cron:**
```bash
curl -X POST http://localhost:3005/api/scheduler/run-now
```

**Change report interval to weekly at runtime:**
```bash
curl -X PUT http://localhost:3005/api/scheduler/interval \
  -H "Content-Type: application/json" \
  -d '{"interval": "weekly"}'
```

**Connect to MongoDB directly:**
```bash
docker exec -it mongo mongosh sports-store-logs
db.eventlogs.find().sort({ createdAt: -1 }).limit(10)
```
